# Grok 视频 API 与系统 openai-video 端点实现对比分析

## 1. 概述

系统通过 openai-video 统一端点（`/v1/videos`）对 Grok 视频 API（grok-imagine-video）进行了全面适配。整体采用异步任务模型：客户端提交视频生成请求后获得任务 ID，通过轮询获取任务状态，完成后下载视频。当前实现已覆盖 Grok 视频 API 的核心功能，包括文生视频、单图/多图图生视频、Remix、任务状态查询和视频内容下载。

系统通过 `relay/channel/task/aigc/` 适配器将请求转发至上游 Grok（Grok2API）网关，并将响应转换为 OpenAI 兼容格式返回。

## 3. 已完善的功能（本次新增）

### 3.1 JSON/Multipart 图像引用字段统一

- **问题**：Grok API 对单图和多图使用不同字段名，且 JSON 和 Multipart 格式字段名也不同
- **实现**：`TaskSubmitReq` 结构同时支持四个字段：
  - `image_reference`（JSON 单图）→ `ImageReference *ImageURLRef`
  - `image_references`（JSON 多图）→ `ImageReferences []ImageURLRef`
  - `input_reference`（Multipart 单图）→ `InputReference string`
  - `input_references`（Multipart 多图）→ 多文件上传
- **效果**：客户端可以按 Grok 原始文档的字段名直接调用，无需适配

### 3.2 quality 响应字段补充

- **问题**：Grok API 返回的视频任务响应中包含 `quality` 字段（如 `"720p"`），原系统 `OpenAIVideo` 结构未定义
- **实现**：上游返回的原始 JSON 通过 `ConvertToOpenAIVideo` 直接透传（仅替换 ID），因此 quality 字段自然保留
- **效果**：客户端可以获取到完整的上游响应信息

### 3.3 Base64 data URI 混用支持

- **问题**：`image_url` 字段既可以是 HTTP URL，也可以是 `data:image/<type>;base64,...` 格式
- **实现**：`ImageURLRef.ImageURL` 为字符串类型，直接透传至上游，不做格式限制
- **效果**：支持同一请求中 URL 和 base64 data URI 混合使用（如多图场景中一张用 URL、一张用 base64）

## 4. 未实现/差异功能

### 4.1 `/v1/chat/completions` 中的 video_config 支持

- **Grok API**：支持通过 `/v1/chat/completions` + `video_config` 参数生成视频，`messages` 中的 `image_url` 块自动作为参考图
- **系统现状**：未实现该路径，视频生成仅通过 `/v1/videos` 端点
- **影响**：不支持通过 Chat Completions SDK 调用视频生成
- **优先级**：中 — 主要影响 SDK 集成体验

### 4.2 多图 4/3 倍率计费

- **Grok API**：多图（≥2 张参考图）在单图/文生价格基础上 × 4/3（向下取整）
- **系统现状**：`EstimateBilling` 中仅计算 `seconds` 和 `size` 维度的倍率，未区分单图/多图
- **影响**：多图请求的计费可能与 Grok 原始定价不一致
- **优先级**：高 — 直接影响计费准确性

### 4.3 流式视频生成

- **Grok API**：`/v1/chat/completions` 设置 `stream: true` + `video_config` 后，服务端通过心跳保持连接，客户端无需显式轮询
- **系统现状**：视频生成仅支持异步轮询模式
- **影响**：客户端需自行实现轮询逻辑
- **优先级**：低 — 轮询模式已能满足需求，且避免了长连接超时问题

### 4.4 resolution_name / preset 参数透传

- **Grok API**：支持 `resolution_name`（480p/720p）和 `preset`（fun/normal/spicy/custom）参数
- **系统现状**：这两个字段未在 `TaskSubmitReq` 中显式定义，但由于 `BuildRequestBody` 直接透传原始 JSON body，实际上这些字段会被传递到上游
- **影响**：功能可用，但缺少服务端校验
- **优先级**：低

## 5. 后续优化建议（按优先级）

| 优先级 | 建议 | 说明 |
| :-- | :-- | :-- |
| P0 | 补充多图 4/3 倍率计费逻辑 | 在 `EstimateBilling` 中检测 `ImageReferences` 数量 ≥2 时乘以 4/3 系数 |
| P1 | `TaskSubmitReq` 增加 `resolution_name` 和 `preset` 字段 | 显式定义并在服务端做取值范围校验 |
| P1 | 添加 `seconds` 合法值校验 | 仅允许 6/10/12/16/20，非法值提前拒绝 |
| P2 | 支持 `/v1/chat/completions` + `video_config` 路径 | 通过中间件识别视频模型并转发至视频任务流程 |
| P2 | `OpenAIVideo` 增加 `quality` / `prompt` 响应字段 | 虽然 `ConvertToOpenAIVideo` 透传了完整 JSON，但显式定义有助于类型安全和文档生成 |
| P3 | 支持流式视频生成（SSE 心跳） | 低优先级，轮询模式已满足需求 |
| P3 | 增加任务过期清理机制 | 对超时未完成的任务自动标记失败并退费 |
